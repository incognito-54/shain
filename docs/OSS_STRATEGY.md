# OSS業務自動化ツール量産戦略

Shain(マルチエージェント基盤)を製造ラインにして、業務自動化ツールをOSSとして量産するための戦略。

---

## 1. なぜ勝ち目があるか

- **日本のSaaSはOSSエコシステムが薄い**。クラウドサイン、freee、マネーフォワード、SmartHR、kintone、弁護士ドットコム系などはAPIを公開しているのに、英語圏(Zapier/n8n中心)からは無視されがちで、公式SDK以外のOSSツールがほぼ無い
- **需要は確実にある**。「SaaS間の転記」は日本中のバックオフィスが毎日手作業でやっている
- **エージェントで限界費用が下がった**。1本作るコストが人力の1/10なら、ニッチすぎて誰も作らなかったツールが採算に乗る

## 2. ポジショニング: 「1ツール=1ジョブ」

Zapier/n8nのような汎用基盤とは戦わない。**特定の1業務を1コマンドで解決する小さなCLI**に絞る。

- ✅ `npx cloudsign-to-sheets` — クラウドサインの締結済み契約をスプシに転記
- ✅ `npx freee-receipt-sync` — レシート画像をfreeeに一括登録
- ❌ 「あらゆるSaaSを繋ぐプラットフォーム」(死亡フラグ)

小さいほど: 作るのが速い / READMEが書きやすい / 検索に引っかかる / 壊れにくい。

## 3. リポジトリ戦略

**個別リポ + テンプレートリポ**方式を推奨。

```
github.com/<you>/automation-tool-template   ← 雛形(CI/README構成/リリース設定込み)
github.com/<you>/cloudsign-to-sheets        ← ツール1
github.com/<you>/freee-receipt-sync         ← ツール2
...
```

- 個別リポの理由: ツール名でのGitHub/npm検索に直接ヒットする、スターが分散せず「このツールが欲しい人」に刺さる、1本壊れても他に波及しない
- テンプレートリポが量産の要。**ここに品質を作り込めば全ツールに複製される**

## 4. 品質基準(全ツール共通のDefinition of Done)

テンプレートに焼き込む。これを満たさないものは公開しない。

- [ ] `npx <tool-name>` の一発実行で動く(グローバルインストール不要)
- [ ] 設定は `.env` または引数のみ。`.env.example` 同梱
- [ ] `--dry-run` オプション(書き込み系は必須。信頼の源泉)
- [ ] README: 30秒で価値がわかる構成(何が自動になるか→GIF→3ステップ導入)。日本語メイン+英語サマリ
- [ ] LICENSE: MIT
- [ ] テスト + GitHub Actions CI(バッジをREADMEに)
- [ ] エラーメッセージが日本語で親切(利用者は非エンジニアのバックオフィス担当も想定)

## 5. ⚠️ LLM依存ツールの設計原則(重要)

PDF読み取りなどでClaudeを使うツールをOSS配布する場合:

- **BYOK(Bring Your Own Key)設計にする**。利用者が自分の `ANTHROPIC_API_KEY` を入れる方式
- **自分のサブスク認証(Claude Codeログイン)を配布物に組み込むことはできない**(規約上、第三者提供にはAnthropicの承認が必要)。開発時にShainで作るのはOK、成果物に埋め込むのはNG
- LLMなしで済む処理(API叩いて転記するだけ等)は使わない。依存が減り、無料で使えるツールは広まりやすい
- LLMが本質的に必要な部分(非定型PDFの項目抽出など)だけに限定し、READMEに概算コスト(例: 1契約あたり約$0.01)を明記

## 6. 量産パイプライン(Shainの使い方)

1ツール=1セッションで、工程を固定する:

```
① 企画    : 対象業務・入出力・完了条件を自分で1段落書く(ここだけは人間の仕事)
② 調査    : researcherに対象SaaSのAPI仕様・認証方式・制約を調査させ docs/research.md に保存
③ 実装    : テンプレートをcloneし、coderに調査結果を渡して実装させる
④ 検証    : reviewerにdry-run含む動作検証+READMEどおりに導入できるかチェックさせる
⑤ 文書    : writerにREADME(日英)とZenn向け紹介記事の下書きを書かせる
⑥ 公開    : 自分が最終確認してpush・npm publish(公開判断だけは必ず人間)
```

依頼文の型は `docs/prompts/` にストックし、2本目以降は差分だけ書く。
**慣れれば1ツール1〜2日**(人力なら1〜2週間)。

## 7. 最初の3本(候補)

| # | ツール | ジョブ | 難易度 |
|---|---|---|---|
| 1 | `cloudsign-to-sheets` | 締結済み契約書の情報をGoogleスプレッドシートに転記 | 中(自分が欲しい=最高のテストユーザー) |
| 2 | `<saas>-backup` | どれか1つのSaaSのデータを定期CSVエクスポート | 低(読み取りのみで安全) |
| 3 | 請求書PDF→会計ソフト転記系 | 経理の手入力を削減 | 中〜高(LLM抽出、BYOK設計の練習) |

1本目は自分の業務で毎週使うものにする。**自分がヘビーユーザーであることが品質と継続の保証**になる。

## 8. 配布と発見性

- npm publish + GitHubトピック(`automation`, `japanese-saas`, `cloudsign` 等)
- ツールごとにZenn/Qiita記事1本(writerに下書きさせる)。日本語SaaS名で検索する人に直接届く
- 3本溜まったら「日本の業務SaaS自動化ツール集」としてまとめ記事+GitHub Orgに昇格を検討

## 9. やらないことリスト

- 汎用プラットフォーム化(スコープ爆発)
- Web UI(CLIで十分。UIはメンテコストが跳ねる)
- 公開前のエージェント成果物の無検証リリース(信頼は一発で失う)
- 各SaaSのAPI利用規約の未確認公開(特にスクレイピング代替は規約違反リスク)
