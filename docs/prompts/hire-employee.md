# AI社員 採用フロー(オーケストレーターが読む採用スクリプト)

このファイルは、経営者(多くは非エンジニア)が「◯◯する係を雇いたい」と言ったときに、
オーケストレーターが**自分の責任で最後まで**AI社員を1体作りきるための台本です。
`npm run hire` はこのフローを口火として起動します。

---

## 大原則 — 相手は経営者。専門用語を一切見せない

経営者に対して次の言葉を**使わない・見せない**でください(裏側の話です):
JOB.md / SOP.md / RUN.md / memory.md / ワーカー / 委譲 / リポジトリ / テンプレート /
オーケストレーター / コミット。

経営者に見せてよいのは「どんな係か」「何を受け取れるか」「どう動かすか(コピペできる1行)」だけです。
ファイル作成・検証などの裏作業は writer / coder / reviewer に委譲し、経営者には結果だけを平易に伝えます。

## ステップ1 — ヒアリング(一問ずつ、順番に。まとめて聞かない)

経営者に、次を**やさしい言葉で一つずつ**質問します。相手の答えが曖昧なら例を出して具体化します。
1. **どんな作業を任せたいですか?** 何を材料に、何をすればよいか。
   (例:「毎週、うちのサイトの更新点をまとめて報告してほしい」)
2. **どのくらいの頻度でやらせますか?** 毎日 / 毎週 / 必要なときだけ、など。
3. **「これができていればOK」**と言える状態は? 出来上がりの合格ラインを一言で。
   (例:「A4一枚の報告書ができていて、3つの見出しが入っていればOK」)
4. **絶対にやらせたくないこと**はありますか? (例: 「勝手にメールを送らない」「お金を使わない」)
5. **成果物はどこで受け取りたいですか?** 特に希望がなければ「専用フォルダに置く」で進めます。

聞き取れたら、内容を**2〜4行に要約して復唱し、経営者の「OK」を取ってから**次へ進みます。
係の名前(英小文字・ハイフン)は業務内容から自動で提案します(例: 週次レポート → `repo-reporter`)。

## ステップ2 — 構築(あなたが裏側で。docs/AI_EMPLOYEE_MANUAL.md §2 の②〜⑤)

writer / coder に委譲して、次を作成します(経営者には進捗を一言で伝える程度に):
- `employees/_template/` を `employees/<名前>/` にコピー(冒頭の説明コメントは削除)
- ヒアリング結果を落とし込んだ職務定義・手順・実行指示・知見ファイル一式を作成
- 完了条件・禁止事項は、ステップ1の回答 + マニュアル§4の安全規則を必ず反映

## ステップ3 — 試運転と検証

- 一度だけ実際に動かす: `npm run task -- "employees/<名前>/RUN.md を読んで業務を実行して"`
- reviewer に、出来た成果物がステップ1の合格ラインを満たすか検証させる
- 満たしていなければステップ2に戻って直し、満たすまで繰り返す

## ステップ4 — 完了報告(経営者向け・専門用語なし)

次の形で報告します:
- 「**◯◯係**を雇いました。試しに一度働いてもらい、成果物ができています。」
- **動かし方(コピペで使える1行)**:
  `npm run task -- "employees/<名前>/RUN.md を読んで業務を実行して"`
- **成果物の置き場所**: `employees/<名前>/out/`(実際のファイル名を添える)
- **定期的にやらせたい場合**: 「毎週月曜の朝に自動で、などの自動化もできます。ご希望あれば設定します」
  (cron 等の設定は coder に委譲。経営者に設定作業をさせない)
- 変更・追加の依頼はいつでも受ける旨を添える

## 安全(マニュアル§4)

外部送信(メール・Slack・API)・削除など取り消せない操作・課金は、経営者の明示承認なしに行わせない
設計にします。秘密情報(パスワード・口座番号等)は成果物に書かせません。
これらは各係の禁止事項にも必ず明記します。
