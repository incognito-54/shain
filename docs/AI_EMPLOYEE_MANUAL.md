# AI社員 運用マニュアル

このマニュアルは、このマルチエージェント基盤(Shain)の上に「AI社員」を作り、
日々の定型業務を継続的に任せるための手順書です。

対象読者は主にオーケストレーター(対話窓口のエージェント)自身、および
AI社員を新規に立ち上げたい人間の利用者です。オーケストレーターは本マニュアルの
「2. 構築手順」に従って作業をワーカー(coder / writer / reviewer 等)に委譲してください。

## 1. AI社員とは

**AI社員** = 職務記述書(JOB.md)を持ち、特定の業務(例: 週次レポート作成、契約書の
転記、調査代行)を定期実行またはワンショット実行(`npm run task`)で継続的に担当する
ロール型エージェントのことです。

重要な原則: **新しい人格やモデルを作るわけではありません。** このリポジトリの
`src/agents.ts` に定義済みの既存ワーカー(researcher / coder / reviewer / writer /
operator)が、業務ごとに用意した「職務定義ファイル一式」を読み込んで、その役を演じる
方式で実現します。AI社員を1体増やすたびに `src/` を書き換える必要はありません。

### 構成要素(ファイル一式)

各AI社員は `employees/<名前>/` ディレクトリの下に、以下のファイル一式を持ちます。

| ファイル | 役割 |
|---|---|
| `JOB.md` | **職務記述書。** その社員の役割・責任範囲・完了条件・報告形式・してはいけないことを定義する、最も上位の定義ファイル。 |
| `SOP.md` | **業務手順書(Standard Operating Procedure)。** JOB.md で定めた業務を、タスクの種類ごとに「何をどの順番で、何を使ってやるか」まで具体化した手順書。 |
| `RUN.md` | **実行指示書。** `npm run task` から**毎回読み込まれるエントリポイント。** 「JOB.md / SOP.md / memory.md を読んでから業務を実行し、成果物と報告を `out/` に書く」という一連の流れをワーカーに指示する短いファイル。人間やcronが叩くコマンドは常にこのファイルを起点にする。 |
| `memory.md` | **学習の蓄積。** 過去の実行で得た知見・注意点・よくある失敗パターンを積み上げていく場所。次回以降の実行品質を上げるための「その社員の経験」。 |
| `out/` | **成果物と報告の出力先。** 業務の成果物(レポート、CSV、下書きなど)と、実行のたびの報告ファイルがここに溜まっていく。 |

この5点(4ファイル + 1ディレクトリ)が揃っていれば、それはAI社員として機能します。
逆に言えば、これらが揃っていない・内容が曖昧なAI社員は正しく動作しません。

## 2. 新しいAI社員を1体作る構築手順

オーケストレーターが新しいAI社員を1体構築するときに従う、6段階の工程です。
各工程で「何を決めるか」と「何を成果物として残すか」を明確にしてから次に進みます。
オーケストレーター自身はファイルを編集できないため、③④以降の作成作業は
writer / coder に、⑤の検証は reviewer に委譲します。

### ① ヒアリング

人間の依頼者に対して、次の4点を確定させるまで質問します。曖昧なまま次工程に
進まないこと。

- **業務内容**: 何を、どのくらいの頻度で、何をインプットに行うか
- **完了条件**: 何をもって「その回の業務が成功した」と判定するか(機械的に検証
  できる形が望ましい。例:「out/にCSVが生成され、行数が0でないこと」)
- **報告先**: 誰が・どこで(out/内のファイル、Slack通知の要否など)結果を確認するか
- **禁止事項**: 触ってはいけないファイル・外部送信の可否・削除操作の可否など

成果物: ヒアリング結果を要約した箇条書き(この後のJOB.md作成にそのまま使う)。

### ② JD(JOB.md)作成

ヒアリング結果をもとに、writer に `employees/_template/` を `employees/<名前>/` として
コピーし、`JOB.md` を作成することを依頼します。最低限、以下の項目を含めること。

- 役割(この社員は何者か、1〜2文)
- 責任範囲(やること/やらないことの境界)
- 完了条件(①で確定したもの)
- 報告形式(out/にどんな形式のファイルを出すか)
- してはいけないこと(①の禁止事項 + 4章の安全規則を踏まえたもの)

テンプレートをコピーして穴埋めした後、各ファイル冒頭にある「テンプレートの使い方」
説明コメントは削除すること。

成果物: `employees/<名前>/JOB.md`

### ③ SOP.md作成

JOB.md を渡した上で、writer(手順に実装検討が絡む場合は coder と協働)に
`employees/<名前>/SOP.md` の作成を依頼します。JOB.md が「What/Why」なら
SOP.md は「How」です。想定される業務パターンごとに、具体的な手順・使うツール・
参照するファイルのパスまで書き下ろします。

成果物: `employees/<名前>/SOP.md`

### ④ RUN.md作成と試運転

writer に `employees/<名前>/RUN.md` の作成を依頼します。内容は次を満たすこと。

- `employees/<名前>/JOB.md`・`SOP.md`・`memory.md` を読むよう明記する
- 業務を実行し、成果物と実行報告を `employees/<名前>/out/` に書くよう指示する
- 報告ファイルには実行日時・完了条件を満たしたかどうか・次回への申し送りを含めるよう指示する

作成後、実際に1回動かして試運転します。

```bash
npm run task -- "employees/<名前>/RUN.md を読んで業務を実行して"
```

成果物: `employees/<名前>/RUN.md`、および試運転で `out/` に生成された成果物・報告

### ⑤ 検証

reviewer に、④で生成された `out/` の成果物を①のヒアリング結果・JOB.mdの完了条件と
突き合わせて検証させます。依頼例:

```bash
npm run task -- "employees/<名前>/JOB.md の完了条件と、直近の employees/<名前>/out/ の成果物を突き合わせて検証して。過不足があれば重大度つきで報告して"
```

問題があれば②〜④に戻って修正し、問題なしと判定されるまで繰り返します。

### ⑥ 運用開始

検証を通過したら、3章の方法で定期実行(cron等)に登録し、運用を開始します。
運用開始後の最初の数回は報告ファイルを人間が目視確認し、問題なければ
確認頻度を下げていきます。

## 3. 運用方法

### (a) 定期実行のさせ方

AI社員の業務実行は、常に `RUN.md` を読ませる依頼文をワンショットモードで
叩くことで行います。

```bash
npm run task -- "employees/<名前>/RUN.md を読んで業務を実行して"
```

これを cron に登録すれば定期実行になります。例えば毎週月曜9時に
`weekly-reporter` という社員を動かす場合、`crontab -e` で以下のように登録します
(パスは環境に合わせて絶対パスにすること)。

```cron
0 9 * * 1 cd /path/to/shain && /usr/local/bin/npm run task -- "employees/weekly-reporter/RUN.md を読んで業務を実行して" >> employees/weekly-reporter/out/cron.log 2>&1
```

補足:

- `npm run task` は内部で `MULTIAGENT_AUTH=login tsx src/index.ts -p` を実行する
  ワンショットコマンドで、1回実行して終了します。cronとの相性がよい構成です。
- ログイン認証(サブスクリプション課金)を前提にしているため、シェルに無効な
  `ANTHROPIC_API_KEY` が残っていると認証エラーになります。その場合は
  `unset ANTHROPIC_API_KEY` してから実行してください。
- cron環境は対話シェルと `PATH` が異なることが多いため、`npm` や `node` の
  絶対パスを使うか、cron内で `nvm`/`node` の初期化を行うようにしてください。

### (b) 報告の読み方

各回の実行後、`employees/<名前>/out/` に以下が出力されます。

- **成果物ファイル**: 業務そのものの出力(レポート、CSV、下書きなど)。ファイル名や
  形式は SOP.md / RUN.md の指示に従う。
- **実行報告ファイル**: その回の実行結果のサマリ。実行日時、完了条件を満たしたか、
  発生した問題、次回への申し送り事項が書かれている。

確認手順の目安:

1. 実行報告ファイルを開き、完了条件を満たしたと書かれているか確認する
2. 満たしていない、または「問題あり」と書かれている場合は成果物と原文を照合する
3. 継続して同じ問題が出る場合は、SOP.md の手順不足かJOB.mdの完了条件の定義不足を疑い、②③工程に戻って修正する

### (c) memory.md の更新ルール

`memory.md` は「その社員が業務を通じて学んだこと」を蓄積する場所です。RUN.md の
指示に、実行の最後に memory.md への追記を含めておくのが基本です。

- 追記してよいもの: 業務上の注意点、過去にハマった落とし穴とその回避策、
  データソースの癖、判断に迷った際の解決方針など
- 追記してはいけないもの: APIキー・パスワード・トークンなどの秘密情報、
  個人情報・機密性の高い業務データそのもの(要約や気づきの記述にとどめる)
- 追記の形式: 日付つきの箇条書きで、時系列に追記していく(古い内容を勝手に
  書き換えたり削除したりしない。矛盾が出たら「上書き」ではなく「訂正」として追記する)
- 肥大化してきたら、人間が定期的に棚卸しして要点だけに整理し直す

## 4. 安全規則

AI社員(=既存ワーカーがJOB.md/SOP.md/RUN.mdを演じたもの)は、通常のワーカーと
同じ権限・同じ制約の上で動きます。JOB.md や SOP.md にどう書かれていても、
以下は必ず守らせてください。

- **外部送信・ファイル削除・課金が発生する操作は、必ず人間の承認を得てから実行する。**
  メール送信、Slack等への外部通知、ファイル/データの削除、有料APIの呼び出しなどが
  該当する。JOB.md の「してはいけないこと」に必ず明記すること。
- **秘密情報(APIキー・認証情報・パスワード等)は、いかなるファイルにも書かない。**
  JOB.md / SOP.md / RUN.md / memory.md / out/ のいずれにも該当しない。認証はすべて
  環境変数やログイン認証(`MULTIAGENT_AUTH=login`)側に任せ、ファイルには一切残さない。
- **`src/` 配下のコード(ワーカー定義・オーケストレーター本体)は変更しない。**
  AI社員の追加・調整はすべて `employees/` 配下のファイル編集で完結させる。
- **破壊的操作(取り消せない操作)は行わない。** 迷ったら実行せず、その旨を報告して
  人間の判断を仰ぐ。試運転や検証は、本番データではなくサンプル/コピーで行う。
