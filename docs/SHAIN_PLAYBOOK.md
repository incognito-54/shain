# Shain 活用プレイブック — エージェントでバリューを産む方法

このドキュメントは、Shain(このマルチエージェント基盤)を「触って面白いおもちゃ」で終わらせず、
実際の時間削減・成果物生産につなげるための実践ガイドです。

---

## 1. 原則: エージェントの価値はどこから出るか

Shainの価値の源泉は3つだけです。ここから外れた使い方は大体コスパが悪くなります。

| 価値の源泉 | 意味 | 例 |
|---|---|---|
| **並列化** | 人間なら順番にやる作業を同時に走らせる | 3つのライブラリを同時調査、複数ファイルの一斉修正 |
| **完遂力** | 指示したら検証まで含めて最後までやる | 実装→テスト→レビューのループを無人で回す |
| **退屈作業の吸収** | 判断は簡単だが手数が多い作業を肩代わり | 転記、整形、定型文書、ボイラープレート |

### 価値が出るタスクの条件(投げる前のチェックリスト)

- ✅ **完了条件が言語化できる**(「動くこと」「表になっていること」)
- ✅ **結果を機械的に検証できる**(テストが通る、ファイルが存在する、数値が合う)
- ✅ **人間がやると30分以上かかる**(5分の作業を投げると指示コストで赤字)
- ✅ **失敗してもやり直しが効く**(不可逆な操作・本番環境はNG)

### 向かないタスク

- ❌ 要件が自分の中で固まっていないもの(まず自分で5分考えた方が速い)
- ❌ 好みの判断が支配的なもの(デザインの微調整など。往復回数が爆発する)
- ❌ GUI操作が必要なもの(ブラウザ操作ツールは未搭載)

## 2. ユースケース・カタログ(コピペで使える依頼文つき)

### A. 技術調査・意思決定の下ごしらえ 【researcher】

自分でググる代わりに、比較表と推奨案まで作らせる。

> Node.jsでPDFからテキスト抽出するライブラリを3つ調査して。
> 比較軸: 日本語対応 / メンテ状況 / ライセンス / 導入の簡単さ。
> 最後に今回の用途(契約書PDFの解析)での推奨を1つ選び、理由を添えて。
> 結果は docs/research/pdf-libs.md に保存して。

**バリューの出方**: 調査30分〜1時間 → 依頼文1分+待ち数分。しかも結果がファイルに残る。

### B. プロトタイプの一気通貫実装 【coder + reviewer】

「動くところまで」を丸ごと任せる。検証込みで依頼するのがコツ。

> ./tools/csv-merge に、複数CSVを結合して重複行を除去するCLIツールを作って。
> 仕様: 引数でファイル複数指定、--key で重複判定列を指定、結果はstdout。
> サンプルCSVを作ってテストし、reviewerに検証させてから完了報告して。

**バリューの出方**: 半日仕事が1依頼に。「reviewerに検証させて」の一文で品質が安定する。

### C. 業務自動化ツールの量産 【researcher → coder → reviewer】

これが**最も投資対効果が高い使い方**。「毎週n時間かかる手作業」をツール化する。

> クラウドサインからダウンロードした契約書PDFを ./inbox に置いたら、
> 契約相手・締結日・金額・契約期間を抽出してCSVに追記するツールを作って。
> まずPDF解析の方法をresearcherに調査させ、その結果で実装して。
> サンプルPDFがないので、ダミーの契約書PDFを自作してテストして。

**バリューの出方**: ツール自体が資産になり、以後毎週の手作業が消える。
エージェント利用の複利効果はここで生まれる。

### D. コードベースの保守・整備 【coder / writer 並列】

自分では後回しにしがちな「重要だが緊急でない」作業を投げておく。

> このリポジトリの全ソースを監査して:
> ① エラーハンドリング漏れの洗い出し(reviewer)
> ② READMEと実装の乖離チェック(writer)
> を並列でやって、結果を優先度つきリストで報告して。

### E. ドキュメント・レポート生成 【writer】

> 今週やった変更(git log参照)をもとに、非エンジニア向けの進捗報告を
> 300字で docs/weekly/2026-W27.md に書いて。

## 3. 依頼の書き方(出力品質の8割はここで決まる)

ワーカーはこの会話を見られません。オーケストレーターが指示を中継するため、
**依頼文に文脈が入っていないと伝言ゲームで劣化します**。

### テンプレート

```
【目的】何のためか(これがあると判断の質が上がる)
【やること】具体的な作業内容
【完了条件】何をもって完了とするか
【制約】触ってはいけないもの、使うべき技術、出力先
```

### Before / After

❌ 「PDFを読み取るやつ作って」
✅ 「経理が毎月手入力している請求書PDF(./samples に3枚)から、発行元・金額・支払期日を抽出してCSV化するスクリプトを作って。3枚全部で正しく抽出できたら完了。出力は ./out/invoices.csv」

### 効くフレーズ集

| フレーズ | 効果 |
|---|---|
| 「reviewerに検証させてから報告して」 | 検証ループが必ず回る |
| 「並列でやって」 | 独立タスクの同時実行を明示的に促す |
| 「結果は◯◯.mdに保存して」 | 成果物が会話に埋もれず資産化される |
| 「わからない点があれば作業前に質問して」 | 誤った前提での暴走を防ぐ |
| 「頼まれていないことはしないで」 | 過剰実装の抑制 |

## 4. 運用ルーチン(習慣化がバリューの鍵)

単発で思い出したときに使うのではなく、**定位置を作る**と回収額が跳ね上がります。

### おすすめの型: 「朝バッチ」

1. 朝、その日の面倒な作業を2〜3件、依頼文の型で書いて投げる
2. 自分は別の仕事をする(エージェントは勝手に完遂する)
3. 昼に結果を確認、修正指示は追加ターンで
4. **うまくいった依頼文は `docs/prompts/` にストックして再利用**

### 週次: 自動化候補の棚卸し

週の終わりに「今週2回以上やった手作業」を1つ選び、ユースケースCの型でツール化を依頼。
**毎週1つ自動化すれば、3ヶ月で12個の資産**になります。

## 5. コスト / サブスク上限の管理

- サブスク認証(`npm run start:login`)なら従量課金はないが、**Pro/Maxのレート上限を消費**する
- 表示される `$` はAPI換算の参考値。「この依頼は割に合ったか」の目安に使う
- 上限が気になるとき:
  - `MULTIAGENT_MODEL=sonnet npm run start:login` — オーケストレーターをSonnetに(体感品質はほぼ維持、消費は大幅減)
  - 調査だけ・文書だけの軽い依頼はワーカー1体で済むよう小さく投げる
- 逆に「ここぞ」の複雑タスク(大きい実装、難しい調査)はOpusのまま投げる価値がある

## 6. アンチパターン

| アンチパターン | なぜダメか | 代わりに |
|---|---|---|
| 曖昧な依頼を投げて対話で詰める | ターン数×コストが膨らむ | 依頼文テンプレで最初に固める |
| 5分で終わる作業を投げる | 指示+確認コストで赤字 | 自分でやる |
| 結果を検証せず信用する | エージェントは自信満々に間違える | 完了条件+reviewer検証を必須に |
| 本番環境・重要ファイルを直接触らせる | bypassPermissionsで即実行される | 作業用ディレクトリで動かし、自分でデプロイ |
| 成果物を会話の中に放置 | 流れて消える | 必ずファイル出力させる |

## 7. 成長ロードマップ(基盤自体への投資)

使い込んで物足りなくなったら、この順で拡張すると効果的:

1. **プロンプト資産化** — うまくいった依頼文を `docs/prompts/` に貯める(コスト0で効果大)
2. **ワーカー追加** — 自分の業務特化ワーカー(例: `accountant`: 経理書類処理専門)を `src/agents.ts` に追加
3. **カスタムMCPツール** — スプレッドシート書き込み、社内API、Slack通知などを `tool()` + `createSdkMcpServer()` で追加 → 「調査して→スプシに転記」が一気通貫になる
4. **定期実行** — launchd/cron + 非対話モードで「毎朝の定型レポート」などを無人化
5. **バックグラウンドワーカー** — `background: true` で長時間タスクを投げっぱなしに

## 付録: 最初の1週間の練習メニュー

| 日 | 依頼 | 学べること |
|---|---|---|
| 1日目 | 技術調査を1件(ユースケースA) | 依頼文の書き方、結果の粒度感 |
| 2日目 | 小さいCLIツールを1個(B) | 完了条件+検証指示の効果 |
| 3日目 | 並列調査(「3つ同時に調べて」) | 並列化の速さを体感 |
| 4日目 | 失敗を体験(曖昧に投げてみる) | 何が伝わらないかを知る |
| 5日目 | 実業務の手作業を1つツール化(C) | 複利の起点を作る |
